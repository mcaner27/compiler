## Executive Summary

This report documents the development of a complete compiler for the B-Minor programming language. The project implements a multi-stage compilation pipeline that transforms B-Minor source code into x86-64 assembly language. The compiler includes lexical analysis, syntax analysis, semantic analysis (name resolution and type checking), and code generation phases.

---

## 1. Project Overview

### 1.1 Objective
Design and implement a fully functional compiler for the B-Minor language that:
- Performs lexical and syntactic analysis of source code
- Constructs an Abstract Syntax Tree (AST)
- Performs semantic analysis including name resolution and type checking
- Generates x86-64 assembly code

### 1.2 Technology Stack
- **Language:** C
- **Lexer Generator:** Flex (Fast Lexical Analyzer Generator)
- **Parser Generator:** Bison (LALR Parser Generator)
- **Target Architecture:** x86-64 Assembly
- **Build System:** GNU Make
- **Compiler:** GCC

### 1.3 B-Minor Language Features
The B-Minor language supports:
- **Data Types:** integer, boolean, char, string, void
- **Variables:** Global and local variable declarations with optional initialization
- **Control Flow:** if/else statements, for loops, while loops
- **Functions:** Function declaration and invocation with parameters and return values
- **Operators:** Arithmetic, relational, logical, assignment, increment/decrement, exponentiation
- **Arrays:** Array declaration and indexing
- **I/O:** Print statement for console output

## 2. Compiler Architecture

### 2.1 Compilation Pipeline

```
B-Minor Source Code
         ↓
    [LEXICAL ANALYSIS]  → Tokens
         ↓
    [SYNTAX ANALYSIS]   → AST
         ↓
    [NAME RESOLUTION]   → Symbol Table
         ↓
    [TYPE CHECKING]     → Type Validation
         ↓
    [CODE GENERATION]   → x86-64 Assembly
```

### 2.2 System Components

#### 2.2.1 Lexical Analysis (Scanner)
**File:** `src/scanner.flex`  
**Purpose:** Convert source code into a stream of tokens

**Features:**
- Recognizes reserved keywords (if, else, for, while, function, return, print, etc.)
- Identifies identifiers, numeric literals, string literals, and character literals
- Handles operators: arithmetic, logical, relational, assignment
- Manages whitespace and comments
- Generates appropriate error tokens for invalid input

**Token Types Recognized:**
- Keywords: TOKEN_IF, TOKEN_ELSE, TOKEN_FOR, TOKEN_WHILE, TOKEN_FUNCTION, etc.
- Identifiers: TOKEN_IDENTIFIER
- Literals: TOKEN_INTEGER_LITERAL, TOKEN_STRING_LITERAL, TOKEN_CHARACTER_LITERAL
- Operators: TOKEN_ADD, TOKEN_SUBTRACT, TOKEN_MULTIPLY, etc.
- Delimiters: TOKEN_LPAREN, TOKEN_RPAREN, TOKEN_LBRACE, TOKEN_RBRACE, etc.

#### 2.2.2 Syntax Analysis (Parser)
**File:** `src/parser.y`  
**Purpose:** Validate grammatical structure and build Abstract Syntax Tree

**Key Features:**
- LALR(1) parser generated by Bison
- Defines BNF grammar for B-Minor language
- Handles operator precedence and associativity
- Constructs AST nodes for programs, functions, statements, and expressions
- Error recovery mechanisms for syntax errors

**Grammar Elements:**
```
program         → declaration_list
declaration_list → declaration | declaration_list declaration
declaration      → type identifier ; | function_decl
function_decl   → identifier : function return_type param_list_opt = block
type            → integer | boolean | char | string | void
statement       → return_expr ; | print print_list ; | if expr block
                | if expr block else block | for expr block
```

#### 2.2.3 Abstract Syntax Tree (AST)
**File:** `src/ast.h`, `src/ast.c`  
**Purpose:** Represent program structure in a tree format for semantic analysis

**Node Types:**
```c
typedef enum {
    NODE_PROGRAM,      // Root node
    NODE_DECLARATION,  // Variable and function declarations
    NODE_FUNCTION,     // Function definition
    NODE_STATEMENT,    // Control flow and expressions
    NODE_EXPRESSION,   // Operations and values
    NODE_TYPE          // Type information
} node_type;
```

**Data Structure:**
```c
typedef struct ASTNode {
    node_type type;
    type_kind type_kind_val;        // integer, boolean, char, etc.
    stmt_kind stmt_kind_val;        // return, print, if, for, etc.
    expr_kind expr_kind_val;        // arithmetic, logical operations
    char *name;                     // Identifier names
    char *string_value;             // String and character literals
    int int_value;                  // Integer constants
    struct ASTNode *left;           // Left child (usually left operand)
    struct ASTNode *right;          // Right child (usually right operand)
    struct ASTNode *third;          // Third child (for ternary operations)
    struct ASTNode *next;           // Sibling nodes (for lists)
} ASTNode;
```

**AST Operations:**
- `create_node()`: Creates new AST nodes
- `print_ast()`: Displays tree structure for debugging
- `print_pretty()`: Generates formatted source code representation

#### 2.2.4 Symbol Table and Scope Management
**File:** `src/symbol.h`, `src/symbol.c`  
**Purpose:** Track variable declarations and manage scope

**Data Structures:**
```c
typedef struct Symbol {
    char *name;           // Variable/function name
    ASTNode *node;        // Reference to declaration node
    struct Symbol *next;  // Hash chain for collisions
} Symbol;

typedef struct Scope {
    Symbol *symbols;      // Hash table of symbols
    struct Scope *parent; // Parent scope for nested scopes
} Scope;
```

**Key Functions:**
- `scope_create()`: Create new scope (for functions, blocks)
- `scope_destroy()`: Free scope resources
- `scope_bind()`: Add symbol to scope (with duplicate checking)
- `scope_lookup()`: Search for symbol in current and parent scopes
- `scope_lookup_current()`: Search only in current scope

#### 2.2.5 Name Resolution
**File:** `src/resolve.h`, `src/resolve.c`  
**Purpose:** Validate that all identifiers are declared before use

**Process:**
1. Traverse AST in depth-first order
2. Create scope for each function
3. Bind variable declarations to scope
4. Resolve identifier references to declarations
5. Report errors for undefined symbols and redefinitions

**Error Detection:**
- Undefined variable references
- Duplicate declarations in same scope
- Scope boundary violations

#### 2.2.6 Type Checking
**File:** `src/typecheck.h`, `src/typecheck.c`  
**Purpose:** Ensure type safety and compatibility of operations

**Type System:**
```c
typedef enum {
    CHECK_TYPE_UNKNOWN,   // Unresolved type
    CHECK_TYPE_INTEGER,   // int type
    CHECK_TYPE_BOOLEAN,   // boolean type
    CHECK_TYPE_CHAR,      // char type
    CHECK_TYPE_STRING,    // string type
    CHECK_TYPE_VOID,      // void (no return value)
    CHECK_TYPE_ERROR      // Type error flag
} check_type;
```

**Type Checking Rules:**
- Arithmetic operations require integer operands
- Comparison operations work on comparable types
- Logical operations require boolean operands
- Assignment verifies compatible types
- Return statements match function return type
- Function calls verify argument and return types

**Validation:**
- `expr_type()`: Determines expression type
- `typecheck_ast()`: Validates entire AST
- `typecheck_ast_with_scope()`: Type checking with symbol scope
- Reports type errors without halting compilation

#### 2.2.7 Code Generation
**File:** `src/codegen.h`, `src/codegen.c`  
**Purpose:** Translate AST into x86-64 assembly code

**Target Architecture:** x86-64 (Intel/AMD 64-bit)  
**Assembly Syntax:** AT&T syntax (Intel-compatible format)

**Generated Code Features:**

1. **Arithmetic Operations:**
   ```assembly
   # Integer arithmetic examples
   mov $5, %rax        # Move immediate value to rax
   mov $3, %rbx        # Move immediate value to rbx
   add %rbx, %rax      # Addition
   sub %rbx, %rax      # Subtraction
   imul %rbx, %rax     # Multiplication
   idiv %rbx           # Division
   ```

2. **Comparison Operations:**
   ```assembly
   cmp %rbx, %rax      # Compare rax with rbx
   setl %al            # Set if less than
   setle %al           # Set if less than or equal
   setg %al            # Set if greater than
   movzb %al, %rax     # Zero-extend byte to full register
   ```

3. **Control Flow:**
   - Label generation for branches
   - Conditional jumps for if/else statements
   - Loop management for for/while statements

4. **Function Calls:**
   - Parameter passing via registers/stack
   - Return value in rax register
   - Stack frame management

**Register Usage:**
- `%rax`: Primary accumulator (results, return values)
- `%rbx`: Base register (temporary storage)
- `%rsp`: Stack pointer
- `%rbp`: Base pointer (frame pointer)

### 2.3 Main Execution Flow

**File:** `src/main.c`

The main program executes the complete compilation pipeline:

```
1. Read command-line arguments (input file path)
2. Open and parse input file
3. Display AST structure (tree and pretty-print views)
4. Perform name resolution
   - Report any undefined symbol errors
5. Perform type checking
   - Report any type safety errors
6. If no errors:
   - Generate assembly code
   - Write to output.s
7. Clean up resources
```

```c
// Proposed improvement
#define SYMBOL_TABLE_SIZE 256

typedef struct {
    Symbol *buckets[SYMBOL_TABLE_SIZE];
} SymbolTable;

// Hash function for faster symbol lookup
static unsigned int hash_symbol(const char *name) {
    unsigned int h = 5381;
    while (*name) h = h * 33 + *name++;
    return h % SYMBOL_TABLE_SIZE;
}
```

## 5. Performance Analysis

### 5.1 Current Compiler Performance

**Compilation Stages Complexity:**

| Stage | Algorithm | Complexity | Status |
|-------|-----------|-----------|--------|
| Lexical Analysis | DFA | O(n) | ✓ Optimal |
| Syntax Analysis | LALR Parser | O(n) | ✓ Optimal |
| AST Building | Tree Construction | O(n) | ✓ Optimal |
| Name Resolution | Linear Search | O(n·m) | ⚠ Can improve |
| Type Checking | Tree Traversal | O(n) | ✓ Good |
| Code Generation | Tree Traversal | O(n) | ✓ Good |

**Legend:** n = number of tokens/nodes, m = symbols in scope

### 5.2 Optimization Opportunities

1. **Symbol Table:** Replace linear search with hash table (O(n·m) → O(n))
2. **AST Reuse:** Cache type information to avoid recalculation
3. **Code Generation:** Use instruction selection algorithms for better code
